# ACM 训练赛赛后分析系统 MVP 设计

## 1. 目标

在不改变现有 `1/`、`2/` 等训练目录的前提下，建立一个可长期每天运行的本地分析工具。工具直接从现有 C++ 文件重建训练数据库，保存历史日报，并基于代码和历史记录生成完成情况、算法分布、代码风险、能力画像和具体复习建议。

MVP 优先保证无额外第三方依赖、可重复运行、结果可解释。它不会假装拥有当前目录中不存在的在线判题记录或正式题目名称。

## 2. 已确认的数据事实

- 根目录下的纯数字目录代表训练场次。
- 完整保存的 C++ 文件默认是 AC 代码，用户没有保存 WA 代码的习惯。
- 空文件、只有通用模板或明显未完成的代码属于已尝试但未完成。
- `.cph/*.prob` 仅包含本地样例和本地路径，不能提供正式题名、题目链接或判题状态。
- Git 提交时间和提交说明可作为辅助信息，但不能作为主数据源，因为部分场次尚未提交。
- 同一题可能有多个实现文件，例如 `2/C(DSUbfs).cpp` 和 `2/C(setBFS).cpp`。

## 3. 方案选择

采用“可重建 JSON 快照 + 独立人工覆盖”。

每次运行都会快速枚举训练目录并计算当前快照。当前数据规模很小，全量发现比维护复杂的数据库迁移更可靠。代码分析结果以文件指纹缓存，未变化的文件不重复分析；日报按日期覆盖生成，因此重复运行不会产生重复记录。

暂不采用 SQLite，因为 MVP 的查询规模不需要数据库。也不采用纯 Git 增量，因为未提交文件同样是有效训练数据。

## 4. 项目结构

```text
2026Summer/
├── 1/ ... 10/                 # 原始训练数据，扫描器只读
├── analyzer/
│   ├── __init__.py
│   ├── models.py              # 数据模型和 JSON 转换
│   ├── scanner.py             # 场次、源码发现和题号归一化
│   ├── validity.py            # 有效代码与模板代码判定
│   ├── classification.py      # 算法标签推断
│   ├── quality.py             # C++ 风险规则
│   ├── profile.py             # 长期画像与复习建议
│   ├── storage.py             # 快照、覆盖和缓存读写
│   └── report.py              # Markdown 日报生成
├── config/
│   └── settings.json          # 扫描范围和统计窗口
├── data/
│   ├── sessions.json          # 自动生成的当前训练数据库
│   ├── manual_overrides.json  # 用户维护，自动流程不覆盖
│   └── analysis_cache.json    # 按文件指纹保存分析结果
├── reports/                   # YYYY-MM-DD.md 历史日报
├── tests/                     # unittest 测试
├── main.py                    # 单一命令行入口
└── README.md                  # 使用方法和数据规则
```

原始训练数据与工具生成数据分离。扫描器仅把根目录下名称完全匹配 `^\d+$` 的文件夹视为场次，因此不会把 `data/`、`reports/` 或 `tests/` 误判为训练。

## 5. 扫描和归一化

### 5.1 文件发现

每个数字场次目录只扫描其第一层的 `*.cpp` 文件，不进入 `.cph/` 或 `output/`。文件匹配不区分扩展名大小写。

每个源码版本记录：

- 相对路径；
- 场次编号；
- 文件名和原始题号；
- 归一化题号；
- 本地时区修改时间；
- 物理总行数；
- 非空且非纯注释的代码行数；
- SHA-256 内容指纹；
- 是否有效代码以及判定理由。

### 5.2 题号归一化

文件名去掉 `.cpp` 后，取开头连续的英文字母作为题号并转为大写。例如 `i.cpp` 归一化为 `I`，`C(DSUbfs).cpp` 和 `C(setBFS).cpp` 都归一化为 `C`。

同一场次中归一化题号相同的文件聚合为一道题，所有源码版本都保留。只要任一版本有效，该题的推断状态就是 `AC`；所有版本均无效时才是 `ATTEMPTED`。

无法从文件名提取题号的 C++ 文件仍会记录，使用完整文件名作为稳定题目标识，并附加 `unrecognized_problem_id` 警告，避免静默丢失数据。

### 5.3 场次日期

场次日期取该场所有 C++ 文件中最晚修改时间对应的本地自然日。场次同时保存最早和最晚修改时间，便于识别跨午夜训练。人工覆盖可以为整个场次指定日期。

## 6. 状态推断和人工覆盖

### 6.1 推断规则

- 有实际解题逻辑的源码：`status = AC`、`source = inferred`。
- 空文件、纯模板或明显未完成源码：`status = ATTEMPTED`、`source = inferred`。

MVP 的有效性检查是确定性的：

1. 去除注释、字符串和字符字面量对结构判定的干扰；
2. 空内容判为 `empty`；
3. 没有 `main` 函数判为 `missing_main`；
4. `main` 中只有同步设置、`return 0` 等已知模板语句判为 `template_only`；
5. 出现 `TODO`、`FIXME` 或空的占位函数只作为辅助信号，只有没有实际解题语句时才判为无效；
6. 其余包含输入、输出、控制流、容器操作、函数调用或非样板计算的实现判为有效。

该检查只判断“是否明显是完整解答”，不判断算法是否能通过未知测试。

### 6.2 人工覆盖

`data/manual_overrides.json` 使用 `场次/题号` 作为键，例如：

```json
{
  "schema_version": 1,
  "problems": {
    "10/J": {
      "status": "ATTEMPTED",
      "note": "赛后确认尚未通过"
    }
  },
  "sessions": {
    "10": {
      "date": "2026-07-13"
    }
  }
}
```

允许的状态只有 `AC` 和 `ATTEMPTED`。覆盖存在时，最终状态采用覆盖值且 `source = manual`，同时保留 `inferred_status` 供审计。自动运行会创建缺失的空覆盖文件，但永远不修改已有覆盖内容。

## 7. `sessions.json` 数据模型

顶层结构包含：

```json
{
  "schema_version": 1,
  "generated_at": "2026-07-13T10:30:00-04:00",
  "training_root": ".",
  "sessions": []
}
```

每个场次包含：

- `session_id`：数字场次编号；
- `directory`：相对目录；
- `date`、`started_at`、`ended_at`；
- `problem_count`、`ac_count`、`attempted_count`；
- `problems`：按规范题号排序的问题列表；
- `warnings`：本场非致命数据问题。

每个题目包含：

- `problem_id`；
- `status` 和 `source`；
- `inferred_status`；
- `valid_code`；
- `files`：一个或多个源码版本及其扫描元数据；
- `algorithm_tags` 和标签证据；
- `quality_findings`；
- 可选人工备注。

“未解决题目”只统计已发现且最终状态为 `ATTEMPTED` 的题。由于没有比赛题目总表，MVP 不臆测从未创建文件的题目；报告会明确说明这一限制。

## 8. 算法分类

MVP 使用可解释的代码特征进行多标签分类，不依赖网络：

- DP：`dp` 数组及状态转移特征；
- 图论：邻接表、BFS、DFS、并查集、最短路等特征；
- 数据结构：栈、队列、优先队列、集合、映射、树状数组、线段树；
- 字符串：字符串匹配、哈希、前缀函数、字典树等特征；
- 数学：模运算、快速幂、组合数、质因数、数论函数；
- 几何：点、距离、叉积、面积及浮点几何特征；
- 贪心：排序配合局部选择、优先队列调度等证据；
- 二分：边界循环和 `check` 判定函数特征。

每个标签保存命中的证据规则。证据不足时标记 `未分类`，不强行猜测。近期分布默认统计最近 7 个有训练记录的自然日，可通过配置修改。

## 9. C++ 代码质量分析

MVP 只报告高信号、可定位的静态风险，每条包含规则编号、级别、文件、行号、说明和建议。首批规则覆盖：

- 固定下标访问可能遇到过短容器；
- 大整数乘法在提升类型前溢出；
- 未初始化的局部标量参与使用；
- `map[key]` / `unordered_map[key]` 在只读意图下可能隐式插入；
- 循环边界与容器长度的可疑混用；
- 二次及以上循环的复杂度风险；
- 大容量多维 DP 的内存和时间风险；
- 浮点常量精度、直接判等和整数除法风险；
- 递归 DFS 的栈深风险；
- 可疑魔数、重复定义和常见样式问题。

规则无法证明的内容必须使用“可能”措辞，并附证据，避免把启发式提示描述成确定 bug。同一文件指纹复用缓存；文件变化后重新分析。

## 10. 能力画像和复习建议

画像只使用最终状态：

- 优势：近期与全历史均有多次 AC，且相关高严重度错误较少的标签；
- 薄弱：存在 `ATTEMPTED`、重复质量问题或长期覆盖不足的标签；
- 数据不足时明确输出“证据不足”，不生成虚假优势或弱项。

复习建议必须引用具体历史证据，优先级依次为：

1. 最近重复出现的同类错误；
2. 最近 `ATTEMPTED` 题对应的算法标签；
3. 近期训练分布中明显缺失、但历史上有失败证据的标签。

每条建议包含来源日期、场次、题号和触发原因。例如：“复查 7 月 13 日第 10 场 J：二维边界计算存在可疑缩减，建议手算奇偶尺寸最小样例。”没有具体证据时不输出泛泛建议。

## 11. 日报

`python main.py` 默认分析最新场次日期，也支持 `--date YYYY-MM-DD`。报告保存到 `reports/YYYY-MM-DD.md`，同一天重复运行覆盖该文件。

报告固定包含：

1. 今日训练总结；
2. 完成情况；
3. 算法分类及近期分布；
4. 代码质量发现；
5. 当前能力画像；
6. 明日复习建议；
7. 数据完整性说明。

`python main.py --scan-only` 只重建数据库和缓存，不生成日报，用于初始导入和排查数据。

## 12. 错误处理

- 单个源码读取失败：记录场次警告并继续扫描其他文件；
- 覆盖文件 JSON 无效或包含非法状态：打印明确错误并退出，避免悄悄忽略人工数据；
- 配置缺失：使用内置默认值并创建示例配置；
- `sessions.json` 写入：先写临时文件，再原子替换，避免中断造成半个 JSON；
- 没有发现训练目录或目标日期无记录：返回非零退出码并给出可执行的提示；
- 报告只写工具目录，不修改、移动或编译原始 C++ 文件。

## 13. 测试策略

使用 Python 标准库 `unittest`，避免额外安装。

测试包括：

- 数字目录发现，忽略非训练目录和嵌套生成物；
- 普通题号、大小写题号和多版本题号归一化；
- 空文件、模板文件、完整解答的有效性判断；
- 同题多版本聚合与 AC 推断；
- 人工状态和日期覆盖；
- 跨午夜场次日期选择；
- 算法标签证据；
- 每条代码质量规则的正例和关键反例；
- 缓存命中和文件变化失效；
- JSON 原子写入与无效覆盖拒绝；
- 日报章节、具体证据和幂等覆盖。

最终验证包含全量单元测试、对真实目录执行 `--scan-only`、校验生成 JSON 可重新解析、再生成 2026-07-13 日报并检查统计一致性。

## 14. MVP 非目标和扩展点

MVP 不抓取在线题面、不连接 OJ、不读取不存在的提交历史，也不让语言模型自动改写人工结论。正式题名、链接、OJ 提交导入和可选 LLM 深度分析留作后续适配器；当前 JSON 中的稳定场次/题目标识和规则证据可直接支持这些扩展。
